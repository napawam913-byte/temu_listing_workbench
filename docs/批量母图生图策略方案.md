# 批量母图生图策略方案

状态：方案讨论稿  
日期：2026-06-10  
目标：用更低成本完成商品主图、详情素材图、SKU 图的批量生成，同时保留单张重生、人工替换和导出模板回填能力。

## 1. 背景

当前系统已经支持链接列表商品的图片槽位、插件生图任务、OSS 回填和店小秘 Excel 导出。原有策略更接近“一个槽位一次生图”，例如 8 张商品图就需要 8 次生成。

新策略改为：

```text
待生图槽位进入候选队列
-> 自动凑成四宫格或九宫格母图任务
-> 一次生成母图
-> 程序按固定边界切成多张 1:1 图片
-> 上传 OSS
-> 回填到对应商品、对应槽位、对应 SKU
-> Excel 导出读取最终激活图片
```

核心目标不是固定生成 8 张，而是让系统能够根据实际任务数量自动组批，降低成本。

## 2. 核心原则

1. 图片生成以“槽位任务”为单位，不以“商品”为唯一单位。
2. 同一个商品的图优先组在同一张母图里，风格更统一。
3. 四宫格用于质量优先，九宫格用于成本优先。
4. 主图、组合图、SKU 图比普通素材图优先级更高。
5. 任何 AI 图都可以被人工上传图、原采集图、历史生成图替换。
6. 导出模板只读取每个槽位当前激活图片，不关心图片来源。
7. 主图轮播图和 SKU 预览图必须分开管理，不混在同一个缩略图列表里。

## 3. 生图模式

### 3.1 九宫格经济模式

适用场景：

- 普通商品素材图
- 批量铺货
- 低成本测品
- 同一商品的多张主图/详情图
- 对画质要求不是最高的辅助图

推荐布局：

```text
3 x 3 九宫格
最多承载 9 个槽位任务
```

优点：

- 单次调用产出最多
- 风格统一
- 成本最低

风险：

- 每格分辨率较低
- 多格任务复杂时容易串图
- 不适合非常复杂的组合 SKU

### 3.2 四宫格质量模式

适用场景：

- 主图
- 高点击率图
- 用户点名重生的图
- 组合商品图
- SKU 图
- 较高价值商品

推荐布局：

```text
2 x 2 四宫格
最多承载 4 个槽位任务
```

优点：

- 每格画面更大
- 商品主体更清楚
- 更适合精修和 SKU 图

### 3.3 单张精修模式

适用场景：

- 首图点击率图
- 九宫格/四宫格失败后的兜底
- 用户明确要求某一张图高质量重生
- 复杂组合 SKU 图

特点：

```text
不等待队列
只生成一个槽位
成本高但稳定性最好
```

### 3.4 人工模式

适用场景：

- 用户自己上传图片
- 从采集图片中选择
- 从历史生成图中选择
- 粘贴图片 URL

人工模式不消耗 AI 额度，但仍然进入统一图片槽位系统。

## 4. 分辨率策略

最终切图目标建议统一为：

```text
800 x 800
1:1
JPG 或 WebP
上传 OSS 后使用 HTTP/HTTPS 地址
```

推荐母图尺寸：

| 模式 | 布局 | 推荐母图尺寸 | 理想切图尺寸 |
| --- | --- | --- | --- |
| 四宫格质量模式 | 2 x 2 | 1600 x 1600 或 2048 x 2048 | 800 x 800 或 1024 x 1024 |
| 九宫格经济模式 | 3 x 3 | 2400 x 2400 或 3072 x 3072 | 800 x 800 或 1024 x 1024 |
| 九宫格兼容模式 | 3 x 3 | 2048 x 2048 | 约 682 x 682，需放大到 800 |
| 四宫格兼容模式 | 2 x 2 | 1024 x 1024 | 约 512 x 512，质量偏低，不建议作为正式导出图 |

如果平台只能输出 2K 正方图，九宫格可以先做，但切图后建议执行：

```text
裁剪 -> 去边 -> 放大到 800x800 -> 轻微锐化 -> 上传 OSS
```

## 5. 任务结构

每个待生成图片都是一个独立任务。

```json
{
  "taskId": "img_task_xxx",
  "linkRecordId": "link_xxx",
  "productId": "602813513964771",
  "targetType": "product-main",
  "slotIndex": 1,
  "slotType": "hero-main",
  "sourceImageUrl": "https://example.com/source.jpg",
  "whiteBackgroundImageUrl": "https://oss.example.com/white.jpg",
  "referenceJson": {},
  "prompt": "Panel-specific prompt",
  "priority": "normal",
  "generationMode": "auto-grid",
  "status": "waiting"
}
```

关键字段说明：

| 字段 | 说明 |
| --- | --- |
| taskId | 单张图片任务 ID |
| linkRecordId | 链接列表记录 ID |
| productId | 商品 ID |
| targetType | product-main、product-material、sales-sku |
| slotIndex | 商品图槽位序号或 SKU 序号 |
| slotType | hero-main、effect、lifestyle、detail、comparison 等 |
| sourceImageUrl | 原采集图 |
| whiteBackgroundImageUrl | 白底主体图 |
| referenceJson | 商品结构、SKU 结构、组合关系 |
| prompt | 当前格子的独立提示词 |
| generationMode | auto-grid、four-grid、nine-grid、single、manual |
| status | waiting、batched、generating、splitting、uploaded、failed |

## 6. 母图批次结构

母图批次记录一张大图和它包含的多个 panel。

```json
{
  "batchJobId": "grid_batch_xxx",
  "layout": "3x3",
  "mode": "nine-grid",
  "status": "generating",
  "batchPrompt": "Create one 3x3 contact sheet...",
  "batchImageUrl": "https://oss.example.com/grid.jpg",
  "panelCount": 9,
  "createdAt": "2026-06-10T00:00:00Z"
}
```

每个 panel 结果：

```json
{
  "taskId": "img_task_xxx",
  "batchJobId": "grid_batch_xxx",
  "panelIndex": 1,
  "cropRect": {
    "x": 0,
    "y": 0,
    "width": 800,
    "height": 800
  },
  "resultImageUrl": "https://oss.example.com/crop-1.webp",
  "ossKey": "products/602813/main/01-hero-main.webp",
  "status": "uploaded"
}
```

这样可以追踪任意一张图来自哪张母图、哪个格子、哪条任务。

## 7. 组批规则

### 7.1 默认组批优先级

```text
同一个链接记录
-> 同一个商品
-> 同一个类目
-> 相同 targetType
-> 相同优先级
```

不建议把完全不相关商品强行混到同一张母图中，因为模型容易串货。

### 7.2 九宫格规则

进入九宫格的任务建议满足：

- 商品相同或类目相近
- 普通素材图优先
- 非复杂组合 SKU
- 非用户点名的高优先级首图

触发条件：

```text
等待任务数 >= 9
或用户点击“立即生成九宫格”
或等待时间超过阈值，例如 10 分钟
```

### 7.3 四宫格规则

进入四宫格的任务建议满足：

- 主图
- SKU 图
- 组合图
- 用户点名重生图
- 高优先级任务

触发条件：

```text
等待任务数 >= 4
或用户点击“立即生成四宫格”
或等待时间超过阈值，例如 5 分钟
```

### 7.4 单张规则

触发条件：

```text
用户选择单张精修
或四宫格/九宫格失败后兜底
或复杂组合 SKU 图
```

## 8. 母图提示词结构

母图提示词必须强调“严格网格”和“每格独立”。

基础结构：

```text
Create one single contact sheet image.
Use a strict 3x3 grid with 9 equal square panels.
Each panel is a separate 1:1 ecommerce image.
Use thick clean white gutters between panels.
Do not let products cross panel boundaries.
Do not merge panels.
Do not add panel numbers, labels, watermarks, logos, QR codes, prices, discounts, star ratings, badges, or platform UI.
Keep each product faithful to its source image and SKU structure.
Use consistent lighting and clean Temu-style ecommerce visual quality.

Panel top-left:
{task 1 prompt}

Panel top-center:
{task 2 prompt}

...
```

四宫格则改为：

```text
Use a strict 2x2 grid with 4 equal square panels.
```

注意：

- panel 位置用 top-left、top-center 等文字描述即可。
- 不要求模型在图上写 panel 编号。
- 如果必须加安全文字，必须是短英文功能词，并且不能跨格。
- 组合商品必须说明“购买包含的组件要同时出现”。

## 9. 切图程序策略

### 9.1 基础裁剪

九宫格：

```text
cellWidth = imageWidth / 3
cellHeight = imageHeight / 3
```

四宫格：

```text
cellWidth = imageWidth / 2
cellHeight = imageHeight / 2
```

### 9.2 安全边距

为了避免切到白色分割线或边框，每格内部裁剪时建议去掉 2%-4% 边缘。

```text
safeMarginX = cellWidth * 0.03
safeMarginY = cellHeight * 0.03
cropX = cellX + safeMarginX
cropY = cellY + safeMarginY
cropW = cellWidth - safeMarginX * 2
cropH = cellHeight - safeMarginY * 2
```

裁剪后统一处理：

```text
中心裁剪为 1:1
resize 到 800x800
转 WebP/JPG
上传 OSS
回填槽位
```

### 9.3 质量检查

初期检查：

- 图片是否下载成功
- 图片尺寸是否足够
- 裁剪结果是否 1:1
- 文件大小是否合理
- OSS 是否上传成功
- URL 是否可访问

后期可加入视觉检查：

- 是否明显空白
- 是否出现文字污染
- 是否有水印/logo
- 是否疑似跨格
- 是否和任务商品不一致

## 10. 前端交互方案

### 10.1 链接列表槽位展示

每个链接记录展示商品图槽位：

```text
图1 主图        已完成    预览 / 替换 / 重生
图2 效果图      待生成    加入队列 / 手动上传
图3 场景图      生成中    查看批次
图4 细节图      失败      重试 / 单张精修
```

### 10.2 重生入口

用户点击某个槽位“重生”时，可以选择：

```text
经济队列：加入九宫格
质量队列：加入四宫格
立即精修：单张生成
手动替换：上传或选择已有图
```

默认建议：

```text
普通图 -> 经济队列
主图/SKU图 -> 质量队列
用户强制 -> 立即精修
```

### 10.3 候选队列页面

后续可以增加一个“生图队列”视图：

```text
待生成 17 张
九宫格可生成 9 张
四宫格可生成 4 张
剩余 4 张等待

[立即生成九宫格]
[立即生成四宫格]
[全部改为单张精修]
```

### 10.4 图片来源切换

每个槽位支持来源：

```text
AI 母图切出
AI 单张生成
原始采集图
人工上传图
历史版本图
```

当前激活图才参与预览和导出。

## 11. SKU 图策略

SKU 图必须和商品主图分开管理。

普通单 SKU：

```text
SKU 图可以使用原 SKU 图
也可以进入四宫格/九宫格生成
```

组合 SKU：

```text
SKU 1：只显示 SKU 1 买家收到的东西
SKU 2：只显示 SKU 2 买家收到的东西
SKU 3：组合 SKU 必须显示所有组合组件
```

复杂组合 SKU 建议优先走四宫格或单张精修，不建议放入普通九宫格混批。

## 12. 导出模板映射

导出时不直接读取原始采集图，而是读取最终激活图片。

| 导出字段 | 来源 |
| --- | --- |
| 轮播图 | 商品主图槽位中激活的 product-main/product-material，最多 10 张 |
| 产品描述图 | 可复用轮播图，或使用单独描述图槽位 |
| 预览图 | SKU 当前激活图片，顺序跟 skuEntries 一致 |
| SKU 顺序 | 前端 SKU 顺序调整后的 skuEntries |

如果 AI 图没有生成完成：

```text
优先使用当前激活图
没有激活图 -> 使用原采集图
仍没有 -> 留空或导出前提示
```

## 15. 落地阶段

### 阶段一：同商品九宫格

目标：

```text
同一个链接记录的商品图槽位
-> 生成九宫格母图
-> 切图
-> 回填图1-图9
```

先不处理跨商品混批，不处理复杂 SKU。

### 阶段二：四宫格重生队列

目标：

```text
用户点某张图重生
-> 进入四宫格候选队列
-> 满 4 张或手动触发生成
-> 切图回填
```

### 阶段三：SKU 图与组合图

目标：

```text
SKU 图独立任务
组合 SKU 识别组件
复杂组合优先四宫格或单张精修
```

### 阶段四：人工替换与版本管理

目标：

```text
每个槽位保留多版本
用户可以切换激活图
导出只读取激活图
```

### 阶段五：质量检测

目标：

```text
自动检测空图、尺寸异常、OSS 失败、疑似水印/文字污染
失败任务进入重试队列
```

## 16. 风险与约束

| 风险 | 解决方式 |
| --- | --- |
| 母图格子不规整 | prompt 强制 strict grid + 白色 gutters + 程序安全边距裁剪 |
| 商品跨格 | prompt 明确 no overlap/no crossing boundaries |
| 不同商品串图 | 优先同商品组批，跨商品组批限制类目 |
| 2K 九宫格切出尺寸不足 | 切后放大，或使用 2400/3072 母图 |
| SKU 组合不准确 | 复杂 SKU 走四宫格或单张 |
| 敏感词/违规文字 | prompt 限制 + 敏感词库过滤 + 人工预览 |
| AI 图质量不稳定 | 单张精修兜底 + 人工替换 |

## 18. 模块任务与跨链接风格锁

这一节用于解决一个关键问题：同一张九宫格母图里可能不只包含一个链接的图片任务。比如用户正在重生多个商品的单张图，系统为了节省成本，希望把这些零散任务凑成一张九宫格生成。此时不能只用一个全局提示词，否则不同商品之间会串风格、串元素，切图后也不像各自商品详情页的一部分。

因此，最终生图逻辑应从“固定 8 张主图”升级为“视觉任务包 + 模块任务 + 链接级风格锁”。

### 18.1 视觉任务包

一个视觉任务包代表一次业务上的生图需求。它可以包含 1 个模块，也可以包含 4 个、5 个、8 个或 9 个模块。

```json
{
  "visualTaskId": "visual_task_xxx",
  "linkRecordId": "link_xxx",
  "taskType": "product_gallery",
  "generationMode": "auto",
  "styleProfileId": "style_xxx",
  "modules": []
}
```

常见任务包类型：

| 类型 | 说明 |
| --- | --- |
| product_gallery | 商品主图组，例如 5 张、8 张或 9 张商品图 |
| sku_gallery | SKU 图片组 |
| combo_gallery | 组合销售图片组 |
| single_refine | 单张精修 |
| mixed_batch | 多个链接的模块混合组批 |

### 18.2 模块任务

模块任务代表最终要产出的一张图。前端槽位、SKU 图、精修图、组合图都可以统一成模块。

```json
{
  "moduleId": "module_xxx",
  "visualTaskId": "visual_task_xxx",
  "linkRecordId": "link_xxx",
  "targetSlotId": "main-3",
  "targetType": "main_image",
  "imageKind": "detail-texture",
  "moduleTitle": "细节材质图",
  "inputImages": ["white_background_url"],
  "skuInfo": null,
  "prompt": "",
  "resultImageUrl": "",
  "status": "pending"
}
```

模块任务的好处是足够灵活：

- 用户要图生图 5 张，就是 1 个视觉任务包 + 5 个模块。
- 用户要重生第 3 张图，就是 1 个视觉任务包 + 1 个模块。
- 用户要 SKU 图，就把 SKU 模块加入任务包。
- 用户要组合出售图，就把多个商品输入图和组合关系写入模块。
- 用户人工替换图片，也可以直接绑定到模块结果，不必进入 AI 队列。

### 18.3 风格锁 StyleProfile

画风一致性不应该写散在每个模块里，而应该抽成链接级风格锁。

每个链接记录至少应该有一个 `styleProfile`：

```json
{
  "styleProfileId": "style_xxx",
  "linkRecordId": "link_xxx",
  "baseReferenceImageUrl": "white_background_or_approved_main_image",
  "approvedReferenceImages": [],
  "productIdentity": "真实商品结构、材质、颜色、数量",
  "visualTone": "bright high-impact US Temu-style ecommerce",
  "backgroundRule": "colorful but clean, product-focused, no clutter",
  "lightingRule": "bright natural light, soft commercial shadows",
  "cameraRule": "close product-focused composition",
  "colorPalette": ["warm white", "soft pink", "fresh green"],
  "textRule": "short safe English text only, no price, no discount",
  "forbiddenElements": ["logo", "watermark", "QR code", "price", "discount", "medical claim", "best", "guaranteed"]
}
```

风格锁的状态建议：

| 状态 | 含义 |
| --- | --- |
| missing | 尚未生成风格要求 |
| draft | 已生成但未确认 |
| locked | 已确认，可用于批量生图 |
| stale | 商品结构或 SKU 变更后需要重新确认 |

批量生成前，系统应优先检查模块所属链接是否已有 `locked` 状态的风格锁。

### 18.4 同链接九宫格

如果九宫格里的 9 个模块都属于同一个链接，处理最简单。

流程：

```text
链接 A
-> 生成或读取链接 A 的 styleProfile
-> 生成 9 个模块提示词
-> 组为 1 个九宫格母图任务
-> 生成母图
-> 切割 9 张
-> 回填到链接 A 的对应槽位
```

这种模式统一性最好，适合一个商品的一整套主图。

### 18.5 多链接混合九宫格

如果九宫格中包含多个链接的模块，必须使用两层风格约束：

1. 全局批次风格：保证整张母图都是 Temu 电商图质感。
2. 链接级风格锁：保证每个格子仍然属于自己的商品详情页风格。

混合九宫格提示词应按格子写清楚：

```text
Create one strict 3x3 grid image. Each cell is an independent ecommerce image.
All cells share a bright high-impact US Temu-style marketplace quality.
Do not mix products between cells. Do not let objects cross cell boundaries.

Cell 1:
Link A. Style profile: soft pink garden gift style, bright natural light.
Task: impact main image for cherry blossom bonsai seeds.

Cell 2:
Link B. Style profile: clean home pet feeding style, white and light blue background.
Task: main image for stainless pet bowl.

Cell 3:
Link C. Style profile: warm jewelry accessory style, gold metal texture, cream background.
Task: detail image for resin initial keychain.
```

混合九宫格进入条件：

| 条件 | 规则 |
| --- | --- |
| 每个链接都有 styleProfile | 必须为 locked |
| 商品相似度过高 | 不建议混合，避免串货 |
| 复杂 SKU 组合图 | 不建议混合，优先单张或四宫格 |
| 结构图、尺寸图、对比图 | 谨慎混合，优先四宫格 |
| 普通视觉图、场景图、细节氛围图 | 可以混合九宫格 |

### 18.6 调度器落地规则

调度器不直接按“商品”组批，而是按“模块任务”组批。

推荐规则：

```text
同一链接 5-9 个模块
-> 优先九宫格

同一链接 2-4 个模块
-> 优先四宫格

多个链接混合模块
-> 检查每个链接 styleProfile
-> 全部 locked 才允许混合
-> 相似度高或结构复杂则降级为四宫格/单张

单张重要主图、复杂组合 SKU、结构准确图
-> 单张优先
```

如果模块等待组批时间超过阈值，可以自动降级：

```text
等待九宫格超过 10 分钟
-> 若已有 4 个可组模块，降级四宫格
-> 若用户选择立即生成，单张执行
```

### 18.9 最终判断

前端永远只关心：

```text
模块最终有没有图
图绑定到哪个商品槽位或 SKU 槽位
当前激活版本是哪一张
```

后端调度器才关心：

```text
这些模块是单张生成
还是四宫格生成
还是九宫格混合生成
是否允许跨链接混合
是否需要先生成风格锁
```

这样可以保证系统既灵活又省钱。后续无论接 GPT API、飞书、插件还是 ComfyUI，都只是替换执行器，不需要推翻前端槽位和任务包结构。

## 19. 最终方案总结

最终图片系统应支持四条路径：

```text
九宫格经济生成
四宫格质量生成
单张精修生成
人工上传/选择已有图
```

默认走：

```text
九宫格铺量
四宫格补质量
单张做关键图
人工兜底
```

这套方案的核心价值是：用候选队列把零散图片需求聚合成母图任务，用程序切图保证成本可控，用版本管理保证结果可回退，用最终激活图保证导出稳定。
