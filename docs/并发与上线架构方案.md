# Temu 选品上架工作台并发与上线架构方案

更新时间：2026-06-11

## 1. 背景

当前项目在本地开发阶段使用 SQLite 作为数据库。SQLite 适合单人、本地、轻量测试，但本项目已经出现多处并发写入场景：

- 前端链接列表自动保存。
- 1688 插件/采集侧边栏回传数据。
- 云启机器人批量导入商品。
- 生图任务创建、规划、生成、切图、回填。
- 登录态校验更新 `last_seen_at`。
- 后续多用户同时使用同一套线上系统。

本地已出现过：

```text
sqlite3.OperationalError: database is locked
```

原因不是后端没有启动，而是 SQLite 文件级写锁在高频并发写入时发生冲突。

## 2. 当前本地临时缓解方案

当前已经在本地后端做了三类缓解，目标是让开发阶段更稳定。

### 2.1 SQLite 等待锁

数据库连接增加 `busy_timeout`：

```text
SQLITE_BUSY_TIMEOUT_MS = 30000
```

作用：当数据库暂时被其他写事务锁住时，当前请求等待一段时间，而不是立刻报错。

### 2.2 启用 WAL

初始化数据库时启用：

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
```

作用：提升 SQLite 在读写并发下的稳定性，减少读请求和写请求互相阻塞。

### 2.3 降低登录态写频率

之前每个接口都会更新 `user_sessions.last_seen_at`，这会让所有请求都变成写请求。

当前策略：

- 只有超过 5 分钟才更新一次 `last_seen_at`。
- `last_seen_at` 更新失败不影响业务接口。
- 认证读取成功后优先放行业务请求。

这只是本地临时优化，不是最终上线方案。

## 3. 上线后推荐总体架构

生产环境不建议继续使用 SQLite 扛并发。推荐架构：

```text
浏览器前端
  ↓
FastAPI 后端
  ↓
PostgreSQL：核心业务数据
  ↓
Redis：任务队列 / 缓存 / 幂等锁
  ↓
Worker：生图、切图、OSS 上传、批量导入、插件同步
  ↓
阿里云 OSS：图片文件
```

部署组合：

```text
前端：Nginx / Vercel / 静态对象存储 CDN
后端：FastAPI + Uvicorn/Gunicorn
数据库：PostgreSQL
任务队列：Redis + Celery / RQ / Dramatiq
图片存储：阿里云 OSS
监控：Sentry + 结构化日志
```

## 4. 数据库方案

### 4.1 本地开发

本地可以继续使用 SQLite：

- 启动成本低。
- 方便调试。
- 单人使用问题不大。

但需要接受它不适合高并发写入。

### 4.2 生产环境

生产环境切换 PostgreSQL：

```text
DATABASE_URL=postgresql+psycopg://user:password@host:5432/temu_workbench
```

PostgreSQL 优势：

- 支持行级锁。
- 支持多连接池。
- 支持事务隔离。
- 支持更稳定的批量写入。
- 支持后续多用户、多机器人、多插件同时工作。

### 4.3 ORM 迁移建议

当前项目大量使用原生 SQLite SQL。后续建议引入：

```text
SQLAlchemy 2.x + Alembic
```

迁移步骤：

1. 保留现有 SQLite 逻辑，先抽象数据库访问层。
2. 新功能优先使用 SQLAlchemy。
3. 逐步把核心表迁移到 SQLAlchemy Model。
4. 用 Alembic 管理生产数据库结构变更。
5. 本地使用 SQLite，线上使用 PostgreSQL。

## 5. 任务队列方案

### 5.1 为什么需要任务队列

以下操作不应该在 HTTP 接口里同步执行：

- GPT 图片分析。
- 九宫格母图生成。
- 图片切割。
- 上传 OSS。
- 云启批量导入。
- 1688 批量素材同步。
- Excel 导出前的批量图片处理。

这些任务耗时长、失败率高、可能依赖第三方接口。同步执行会导致：

- 接口超时。
- 数据库连接长时间占用。
- 前端卡住。
- 并发请求堆积。
- 一个模型请求失败影响其他接口。

### 5.2 推荐任务状态

生图任务建议状态：

```text
pending              待执行
analyzing            图片分析中
planned              提示词规划完成
generating_mother    九宫格母图生成中
splitting            切图中
uploading            上传 OSS 中
completed            完成
failed               失败
cancelled            已取消
```

导入任务建议状态：

```text
pending
validating
importing
indexing
completed
failed
```

### 5.3 推荐接口模式

前端创建任务：

```http
POST /api/visual/tasks
```

后端只做：

```text
1. 校验参数
2. 写入任务记录
3. 投递队列
4. 立即返回 task_id
```

Worker 异步执行：

```text
analyze → plan → generate mother → split → upload → update link record
```

前端查询任务：

```http
GET /api/visual/tasks/{task_id}
GET /api/visual/tasks?status=running
```

后续可升级为 WebSocket / SSE 推送进度。

## 6. 前端并发控制

### 6.1 链接列表保存防抖

当前前端容易频繁触发 `POST /api/link-records`。建议：

```text
用户操作后 800ms - 1500ms 防抖保存。
同一条 linkRecord 在保存中不重复提交。
短时间内多次修改合并为一次保存。
保存失败后指数退避重试。
```

示例策略：

```text
第 1 次失败：1 秒后重试
第 2 次失败：3 秒后重试
第 3 次失败：8 秒后重试
之后提示用户手动重试
```

### 6.2 批量保存

不要每条记录都单独请求。优先使用：

```http
POST /api/link-records/batch
```

适用场景：

- 页面初始化后同步本地缓存。
- 插件一次回传多条素材。
- 机器人导入多条商品。
- 批量调整图片槽位。

### 6.3 前端本地缓存策略

本地缓存可以继续作为兜底：

```text
先写 localStorage / IndexedDB
再异步同步后端
后端成功后标记 synced
失败则进入 pendingSync 队列
```

但 UI 文案要区分：

```text
本地已保存，等待同步后端
后端同步成功
后端同步失败，稍后重试
```

不要简单提示“后端未启动”，因为实际可能是数据库锁、网络慢、登录过期或接口报错。

## 7. 插件与机器人同步方案

### 7.1 插件回传

插件不应该高频逐条写入。推荐：

```http
POST /api/plugin/1688/batch
```

请求体：

```json
{
  "source": "1688-extension",
  "items": [],
  "clientBatchId": "plugin_20260611_001"
}
```

后端处理：

```text
1. 校验用户
2. 根据 clientBatchId 幂等去重
3. 批量 upsert 商品/素材
4. 写入导入日志
5. 返回成功/失败明细
```

### 7.2 云启机器人导入

云启机器人可以在本地运行，向云端上传采集结果。

推荐接口：

```http
POST /api/import/yunqi/batch
```

建议一次提交：

```text
100 - 500 条商品
```

不要一条商品请求一次接口。

### 7.3 幂等键

所有机器人/插件写入都要带幂等键：

```text
user_id + source_type + source_product_id
user_id + client_batch_id
user_id + link_record_id + image_slot_id
```

避免网络重试导致重复数据。

## 8. 生图任务并发方案

### 8.1 任务拆分

当前生图逻辑：

```text
1. 图片分析
2. 提示词规划
3. 生成九宫格母图
4. 切割成单图
5. 上传 OSS
6. 回填链接记录
```

上线后每一步都应该是任务状态，而不是一次 HTTP 请求同步跑完。

### 8.2 队列优先级

推荐优先级：

```text
单图精修 > SKU 适配 > 主图多生 > 批量补图
```

原因：

- 单图精修通常是用户正在等结果。
- 主图多生可排队慢慢跑。
- 批量补图可以低优先级夜间处理。

### 8.3 并发限制

对第三方 AI 接口要做限流：

```text
每个用户同时生图任务：1 - 2 个
全局同时生图任务：按 API 配额设置
失败重试：最多 2 - 3 次
```

避免 API 费用失控，也避免 429/413/超时导致任务雪崩。

### 8.4 九宫格任务结果

数据库建议保留：

```text
visual_generation_tasks
visual_generation_modules
visual_generation_artifacts
```

其中：

- `tasks` 存任务整体状态。
- `modules` 存每个格子的目的、提示词、槽位、状态。
- `artifacts` 存母图、切割图、OSS URL、失败日志。

## 9. 数据一致性策略

### 9.1 链接池回填

生图完成后不要直接覆盖所有图片，建议：

```text
生成结果先进入候选图库
用户确认后替换商品图片槽位
也可以设置自动替换策略
```

自动替换时需要记录：

```text
旧图片 URL
新图片 URL
替换时间
操作来源
任务 ID
```

### 9.2 失败恢复

任务失败时保留：

```text
失败阶段
错误信息
请求模型
提示词摘要
参考图
重试次数
```

这样可以从失败阶段继续，而不是整个任务重跑。

## 10. 监控与日志

### 10.1 后端日志

建议记录结构化日志：

```json
{
  "event": "visual_task_failed",
  "taskId": "visual_xxx",
  "userId": "user_xxx",
  "stage": "generating_mother",
  "model": "gpt-image-2",
  "error": "HTTP 413",
  "createdAt": "2026-06-11 14:30:00"
}
```

### 10.2 关键告警

需要监控：

- 数据库连接失败。
- 任务队列堆积。
- AI API 429/413/401。
- OSS 上传失败。
- 插件同步失败。
- 云启机器人导入失败。
- 单用户异常高频请求。

### 10.3 推荐工具

```text
Sentry：异常追踪
Prometheus + Grafana：指标监控
结构化日志文件：本地和小规模部署
云服务日志：正式上线后使用
```

## 11. 分阶段落地计划

### 第一阶段：本地稳定

目标：继续用 SQLite，但减少锁冲突。

已完成：

- SQLite `busy_timeout`。
- WAL。
- 登录态少写库。
- 生图提示词瘦身，减少 413。

建议继续做：

- 前端链接保存防抖。
- 链接池批量保存优先。
- 生图任务执行时减少同步保存链接列表。

### 第二阶段：数据库抽象

目标：为 PostgreSQL 做准备。

任务：

- 引入 SQLAlchemy。
- 抽象商品、链接、任务、用户、配置的数据访问层。
- 保持本地 SQLite 可用。
- 新增 Alembic 迁移脚本。

### 第三阶段：上线数据库

目标：生产环境切 PostgreSQL。

任务：

- 准备 PostgreSQL 实例。
- 配置 `DATABASE_URL`。
- 迁移当前 SQLite 数据。
- 接入连接池。
- 做备份策略。

### 第四阶段：任务队列

目标：生图、导入、上传全部异步化。

任务：

- 部署 Redis。
- 引入 Celery/RQ/Dramatiq。
- 生图任务改成队列执行。
- 云启导入改成队列执行。
- 前端改为查询任务状态。

### 第五阶段：多用户与插件生产化

目标：支持多人使用和插件稳定回传。

任务：

- 每个用户数据隔离。
- API 配置按用户/团队隔离。
- 插件 token 鉴权。
- 插件批量同步。
- 操作日志和审计。

## 12. 当前结论

短期：

```text
SQLite 可以继续本地测试，但需要防抖、批量保存、减少认证写库。
```

中期：

```text
引入 SQLAlchemy，为 PostgreSQL 做迁移准备。
```

长期：

```text
生产环境必须使用 PostgreSQL + Redis 队列。
生图、导入、OSS 上传都应异步执行。
插件和机器人必须批量同步并支持幂等。
```

这套方案可以避免后期上线后因为并发写入、AI 长任务、插件高频回传导致系统不稳定。
