# DxmTemuTerminalRobot 机器人原理

更新时间：2026-06-05

本文只解释 `DxmTemuTerminalRobot` 这个机器人的运行原理，不展开整个 `temu_listing_workbench` 项目的前端、后端、插件等其他模块。

## 1. 机器人定位

`DxmTemuTerminalRobot` 是一个本地桌面自动化机器人，用来辅助操作店小秘 Temu 半托管商品编辑页。

它的核心目标不是“完全替代人工”，而是把重复、固定、容易出错的网页操作自动化，同时保留人工可检查、可停止、可修改、可继续的空间。

机器人主要处理这些事情：

- 连接已经登录店小秘的固定 Chrome。
- 打开或绑定 Temu 商品编辑页。
- 识别页面里的基本信息、产品信息、变种信息、产品描述、运输信息等模块。
- 自动补全必填信息。
- 下载商品图片。
- 删除页面原图。
- 上传本地处理后的图片。
- 填写变种信息里的仓库、库存、申报价格、尺寸、重量等字段。
- 清空并重传产品描述图片。
- 填写运输信息。
- 根据用户选择，进入人工审核模式或直接发布模式。
- 出错、停止或完成时，可通过飞书通知用户。

## 2. 机器人由哪些部分组成

机器人运行时主要由四层组成：

| 层级 | 说明 |
| --- | --- |
| 桌面控制层 | Tk 窗口，负责按钮、配置、停止、飞书测试、人工确认等操作。 |
| 配置状态层 | 本地 JSON 文件，保存仓库、运费模板、图片路径、AI API、飞书 webhook、运行状态。 |
| 浏览器自动化层 | Playwright 通过 CDP 连接 Chrome，对店小秘网页执行点击、输入、上传、读取状态等动作。 |
| 机器人业务层 | 按固定流程执行商品补全、图片处理、仓库选择、发布、错误恢复等逻辑。 |

打包后的主程序是：

```text
dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe
```

源码入口是：

```text
terminal_app.py
```

机器人基座程序是：

```text
work/terminal_app_base.pyc
work/extracted_terminal_app.pyc
```

当前源码 `terminal_app.py` 会加载基座程序，然后覆盖或增强其中的一部分函数。这样可以保留原来已经能跑的 RPA 能力，同时继续修补仓库、图片、停止、配置、发布等新逻辑。

## 3. 启动原理

机器人启动后，先做这些事情：

1. 初始化运行目录。
2. 加载配置文件。
3. 创建 Tk 控制面板。
4. 准备停止事件 `STOP_EVENT`。
5. 注册窗口关闭和进程退出清理逻辑。
6. 等待用户点击“自动补全”“手动菜单”“确认发布”“统一配置”等按钮。

启动后不会立即执行商品流程，真正开始自动化是在用户点击对应按钮之后。

## 4. 配置加载原理

机器人会从本地 JSON 文件读取配置。

主要配置文件：

```text
dist/DxmTemuTerminalRobot/work/state/automation-config.json
```

AI 配置文件：

```text
dist/DxmTemuTerminalRobot/work/state/laozhang-api.json
```

常见配置包括：

- 图片根目录。
- 批量商品数量。
- 是否每次运行自动创建图片子目录。
- 是否填写完成后直接发布。
- 仓库。
- 仓库模板。
- 运费模板。
- 承诺发货天数。
- 飞书 webhook。
- 飞书 secret。
- AI Base URL。
- AI API Key。
- AI 模型。
- 图片处理尺寸、质量和压缩规则。

配置窗口点击“保存配置”后，会写回 JSON。下一次启动机器人时，应该直接恢复上一次保存的配置，不需要重新选择。

## 5. Chrome 连接原理

机器人不是直接打开普通 Chrome 窗口，也不是每次都重新登录店小秘。

它使用固定用户目录：

```text
work/chrome-profile
```

这个目录保存 Chrome 登录态。机器人通过 Playwright 的 CDP 能力连接本地 Chrome 调试端口，例如：

```text
http://127.0.0.1:9222
```

这样做的好处是：

- 店小秘登录态可以复用。
- 用户能看到真实浏览器页面。
- 机器人操作和人工操作发生在同一个页面上下文里。
- 出错时用户可以直接接管页面。

## 6. 页面识别原理

店小秘的 Temu 编辑页不是普通静态表单，而是一个复杂的动态页面。

机器人不会只靠固定坐标操作页面，而是结合以下方式识别模块：

- 页面文本。
- DOM 结构。
- 输入框、按钮、下拉框、图片模块等元素。
- 店小秘模块标题，例如 `基本信息`、`产品信息`、`变种信息`、`产品描述`、`运输信息`。
- 可见元素状态。
- 当前编辑页 URL。

机器人会尽量限定操作范围，例如：

- 产品标题只在 `产品信息` 模块内处理。
- 仓库只在 `变种信息` 模块内处理。
- 运费模板只在 `运输信息` 模块内处理。
- 产品描述图片只在产品描述编辑器中处理。

这样可以降低点错模块、误改字段的概率。

## 7. 自动流程总览

一次完整的商品自动补全流程大致如下：

```text
开始任务
  ↓
连接或绑定当前 Chrome 页面
  ↓
打开店小秘 Temu 编辑页
  ↓
识别页面模块
  ↓
补全基本信息
  ↓
修正产品信息标题
  ↓
下载产品信息图片
  ↓
删除页面原图
  ↓
本地处理图片
  ↓
重新上传产品信息图片
  ↓
填写变种信息
  ↓
处理产品描述图片
  ↓
填写运输信息
  ↓
人工审核或直接发布
  ↓
记录状态 / 通知结果
```

机器人每走一步都会写日志。如果某一步失败，会根据错误类型决定是重试、恢复页面、弹窗让用户选择，还是跳过当前商品。

## 8. 基本信息补全原理

基本信息补全主要处理红星必填项。

它的原则是：

- 只处理当前为空或显示“请选择”的必填项。
- 已经填写好的字段不重复改。
- 敏感规则命中时停止自动填写，交给人工复核。
- AI 只用于判断候选项，不直接操作页面。
- 最终点击、输入、选择由固定 Python RPA 执行。

红线规则包括：

```text
液体
带电
棉
```

如果标题、英文标题、分类、当前属性值或候选项命中这些风险词，机器人会停止相关自动填写，避免误填高风险商品。

## 9. 产品信息图片处理原理

产品信息图片采用“先下载、再删除、再上传”的逻辑。

具体流程：

1. 在 `产品信息` 模块中收集图片链接。
2. 下载图片到本地图片目录。
3. 生成 `manifest.json`，记录下载结果。
4. 只删除已经成功下载的页面原图。
5. 对本地图片进行后处理。
6. 点击店小秘页面里的图片上传入口。
7. 选择 `本地图片`。
8. 上传本地目录里的处理后图片。

这样设计是为了让用户可以在本地图片编辑器中介入：

```text
下载图片 → 删除页面原图 → 人工或程序编辑图片 → 重新上传
```

如果图片处理启用，机器人会把图片处理成符合配置的格式，例如：

- 默认 800 x 800。
- 1:1 方图。
- 白底填充。
- 压缩到指定大小以内。
- 过小图片移入忽略目录。

## 10. 产品描述图片处理原理

产品描述图片不是逐张小心删除，而是优先使用店小秘自己的批量菜单。

目标流程：

1. 打开产品描述编辑器。
2. 检查是否存在图片模块。
3. 如果存在图片模块，点击：

```text
批量操作 → 清空描述 → 清空图片模块
```

4. 如果没有图片模块，则跳过清空。
5. 点击：

```text
批量操作 → 批量传图 → 选择图片 → 本地图片
```

6. 上传本地图片。
7. 点击保存。

这个逻辑可以处理两种页面状态：

- 原来有产品描述图片：先清空，再上传。
- 原来没有产品描述图片：直接上传。

## 11. 变种信息填写原理

变种信息是机器人最重要也最容易失败的模块之一。

机器人会处理：

- 仓库选择。
- 库存。
- 申报价格。
- 尺寸。
- 重量。
- SKU 相关字段清理。

这部分尽量使用固定规则，不依赖 AI。

机器人会按表头和输入框名称定位字段，而不是只看屏幕坐标。这样页面横向滚动、表格列顺序变化时，仍然有机会定位正确字段。

## 12. 仓库选择原理

仓库选择由三部分组成：

1. 配置文本框。
2. 页面多选下拉框。
3. 最终精确校验。

用户配置可以写：

```text
美中仓库，美西仓库，美东仓库
```

页面里可能显示：

```text
美中仓库（其他）
美西仓库（其他）
美东仓库（其他）
```

机器人会把 `（其他）`、`(其他)`、`（其它）`、`(其它)` 当成上下文后缀处理。因此配置中的 `美中仓库` 可以对应页面中的 `美中仓库（其他）`。

但这不是无限模糊匹配。机器人不会因为都包含“美东”就把不同仓库当成同一个仓库。

仓库最终校验必须满足：

- 配置里的仓库全部选中。
- 配置外的仓库全部取消。

例如用户只配置：

```text
美中仓库，美西仓库，美东仓库
```

如果页面实际还勾选了：

```text
煊烊-美东仓（其他）
```

机器人应该把这个模板外仓库取消。如果取消失败，最终校验会报错，而不是假装成功。

## 13. 仓库自检弹窗原理

当机器人找不到配置仓库，或者最终校验发现少选/多选时，会弹出仓库自检窗口。

这个窗口的作用是让用户从页面读取到的真实仓库候选中重新选择。

自检窗口会显示：

- 当前原配置。
- 失败原因。
- 页面可见仓库候选。
- 多选列表。
- 手动补充输入框。
- 是否保存为模板。
- 模板名。
- 保存配置并继续。

用户选择后，机器人会立即：

1. 对仓库去重。
2. 转换成统一格式。
3. 回填主窗口“选择仓库”文本框。
4. 保存到 `automation-config.json`。
5. 使用回填后的仓库继续执行原来的多仓库选择逻辑。

这个弹窗不是替代原选择逻辑，而是修正配置来源。真正点选页面仓库时，仍然走原来的多仓库选择和最终校验流程。

## 14. 发布原理

机器人有两种发布模式。

### 14.1 人工审核模式

如果没有勾选“填写完成后直接发布”，机器人只负责编辑商品。

编辑完成后，当前编辑页会被记录下来。用户人工检查后，再点击“确认发布已编辑页”。

这种模式适合：

- 商品质量要求高。
- 图片需要人工确认。
- 仓库/库存/申报价格需要人工看一眼。
- 不希望机器人自动点发布。

发布已编辑页时，如果某个商品发布失败，机器人会记录失败并跳过当前商品，继续处理后面的商品。

### 14.2 直接发布模式

如果勾选“填写完成后直接发布”，机器人在每个商品编辑完成后立即尝试发布。

这种模式适合：

- 规则已经稳定。
- 商品来源比较统一。
- 允许异常商品失败后跳过。

直接发布失败时，机器人应该记录错误，必要时发飞书通知，但不应该无限卡在同一个商品页面上。

## 15. 停止按钮原理

停止按钮不是强杀浏览器，也不是关闭当前页面。

点击停止按钮后，机器人会：

1. 设置 `STOP_EVENT`。
2. 当前流程在下一个检查点发现停止请求。
3. 抛出可识别的停止异常。
4. 记录停止日志。
5. 必要时发送飞书停止通知。

停止任务后，店小秘页面应该保留给用户人工查看。

也就是说：

```text
停止当前任务 ≠ 关闭网页
停止当前任务 ≠ 关闭 Chrome
停止当前任务 ≠ 杀掉所有浏览器进程
```

这能避免用户正在查看的页面突然消失。

## 16. 窗口关闭和进程清理原理

停止任务和关闭窗口是两件不同的事。

关闭机器人窗口时，机器人需要确保自己的进程不要残留。

关闭窗口时应该做：

- 设置停止事件。
- 结束机器人自身任务线程。
- 清理机器人自己启动的子进程。
- 清理锁文件。
- 退出 Tk 主循环。

关闭窗口时不应该反复尝试关闭已经失效的 Playwright page/context/browser。否则容易出现类似：

```text
Event loop is closed! Is Playwright already stopped?
```

这类日志的意思是：Playwright 的事件循环已经关闭了，但代码还在尝试关闭 page、context 或 browser。对于当前设计，普通停止任务时不需要关闭页面；关闭页面本来就是用户手动决定的事情。

## 17. 飞书通知原理

飞书通知是可选能力。

如果 webhook 为空，机器人会自动认为飞书未启用。

如果配置了 webhook，机器人可以在以下场景发送通知：

- 测试通知。
- 任务失败。
- 用户停止任务。
- 任务成功。

飞书配置保存在本地配置文件中，不写死到源码。

## 18. 错误处理原理

机器人运行过程中可能遇到：

- 页面没加载出来。
- 编辑器没打开。
- 图片模块数量不符合预期。
- 仓库多选状态不一致。
- 商品信息缺失。
- 发布失败。
- 用户点击停止。
- 浏览器连接断开。

处理策略通常分为四种：

| 错误类型 | 处理方式 |
| --- | --- |
| 短暂页面状态问题 | 等待页面状态变化或重试。 |
| 可恢复的模块问题 | 关闭弹窗、恢复页面、重新执行当前步骤。 |
| 配置不匹配问题 | 弹出自检窗口，让用户选择并保存配置。 |
| 商品本身无法发布 | 记录失败，跳过当前商品，继续后续商品。 |

机器人不会把所有错误都简单理解成“程序崩了”。它会尽量判断是页面状态、配置问题、商品问题还是用户停止。

## 19. 打包原理

机器人通过 PyInstaller 打包。

打包脚本：

```text
scripts/package_terminal_exe.py
```

输出位置：

```text
dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe
```

打包时会：

- 把 `terminal_app.py` 打成 onefile exe。
- 收集 Playwright 依赖。
- 收集 tkinter、ttk、filedialog、messagebox 等 GUI 依赖。
- 把基座 pyc 文件放入 exe。
- 把 `work` 目录复制到运行包。
- 忽略 Chrome 缓存、日志、临时文件等无用内容。
- 保留运行包中已有的 `work/state/*.json`，避免覆盖用户配置。

所以用户平时运行的是：

```text
dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe
```

维护源码时主要改：

```text
terminal_app.py
```

然后重新打包。

## 20. 机器人最重要的设计原则

这个机器人的设计原则可以总结成几句话：

1. 页面操作尽量模拟真实人工操作，不直接乱改页面内部状态。
2. 配置必须持久化，用户保存过的仓库、模板、图片目录、API 和 webhook 不应丢失。
3. 图片必须先下载，再删除，再上传本地处理后的图片。
4. 仓库选择必须最终精确校验，不能只靠看起来选中了。
5. 停止任务不关闭页面，关闭窗口才清理机器人自身进程。
6. 直接发布模式下，单个商品发布失败要跳过，不要卡死整个批次。
7. 人工审核模式下，机器人只负责编辑，发布由用户确认。
8. webhook 为空时，飞书通知自动关闭。
9. API Key、webhook、用户配置不写死进源码。
10. 打包时保留用户配置，清理缓存和临时文件。

## 21. 一句话说明

`DxmTemuTerminalRobot` 的本质是：一个通过 Tk 控制面板管理配置、通过 Playwright 连接固定 Chrome、按店小秘页面状态执行真实网页操作，并把图片、仓库、发布、停止和通知做成可控流程的本地 Temu 商品编辑机器人。
