# Temu Listing Workbench 项目原理

更新时间：2026-06-05

## 1. 项目定位

`temu_listing_workbench` 是一个面向 Temu 半托管商品上架、编辑、采集和发布的本地自动化工作台。项目不是单一脚本，而是由前端工作台、后端采集/导出模块、浏览器插件、桌面机器人和本地运行包共同组成。

项目的核心目标是：

- 把 1688、Temu、店小秘等页面中的商品信息采集、整理、补全和发布流程尽量自动化。
- 把需要人工判断的地方保留下来，例如仓库模板选择、商品编辑后人工审核、敏感信息复核。
- 把长期配置保存到本地，避免每次启动都重新填写仓库、运费模板、图片目录、AI API、飞书 webhook 等信息。
- 把源码、配置、运行包和 exe 统一管理，方便后续维护、打包、迁移和清理。

## 2. 总体架构

项目主要由以下部分组成：

| 目录/文件 | 职责 |
| --- | --- |
| `terminal_app.py` | `DxmTemuTerminalRobot` 的主入口和覆盖层源码，负责 Tk 控制面板、配置管理、停止/退出、图片处理、仓库自检、飞书通知、发布策略等增强逻辑。 |
| `work/terminal_app_base.pyc` | 机器人基座程序，包含原始 Playwright RPA 能力。当前 `terminal_app.py` 会加载并覆盖其中部分函数。 |
| `work/extracted_terminal_app.pyc` | 旧版完整程序模块，打包和兼容运行时需要保留。 |
| `scripts/package_terminal_exe.py` | PyInstaller 打包脚本，把机器人打包为 `dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe`，同时复制运行需要的 `work` 目录并保留配置。 |
| `dist/DxmTemuTerminalRobot` | 桌面机器人交付目录，包含 exe、运行时 work、状态配置等。 |
| `frontend` | 前端商品工作台，主要用于商品管理、筛选、规则配置和可视化操作。 |
| `backend` | 后端采集、导出、筛选和 Yunqi 相关模块。 |
| `browser_extension` | 浏览器插件，用于页面采集、前端信号读取和浏览器内辅助能力。 |
| `storage` | 本地数据、模板、导出结果、上传文件、类目库等持久化资料。 |
| `docs` | 项目文档和技术说明。 |

可以把整个项目理解为三层：

1. 数据层：`storage`、`work/state`、导出文件、下载图片、模板配置。
2. 自动化层：`terminal_app.py`、Playwright、浏览器插件、后端采集脚本。
3. 操作层：Tk 桌面控制面板、前端工作台、店小秘网页、飞书通知。

## 3. DxmTemuTerminalRobot 的运行原理

`DxmTemuTerminalRobot` 是一个 Python 桌面自动化机器人。它通过 PyInstaller 打包成 exe，但核心逻辑仍然是 Python。

运行时主要分为五步：

1. 启动 Tk 控制面板。
2. 加载本地配置文件，例如仓库、运费模板、图片目录、AI API、飞书 webhook。
3. 通过 Playwright CDP 连接固定 Chrome。
4. 在店小秘 Temu 商品编辑页上执行自动填写、图片处理、变种信息、产品描述、运输信息等流程。
5. 根据用户选择，进入人工审核模式或直接发布模式。

机器人连接的浏览器不是一次性临时浏览器，而是固定登录目录：

```text
work/chrome-profile
```

这个目录保存店小秘登录态。它不能随便删除，否则需要重新登录。

## 4. 基座程序与覆盖层

当前机器人采用“基座程序 + 覆盖层”的结构。

基座程序位于：

```text
work/terminal_app_base.pyc
work/extracted_terminal_app.pyc
```

覆盖层位于：

```text
terminal_app.py
```

运行时 `terminal_app.py` 会加载基座程序中的函数、类和常量，然后把部分关键函数替换为新的实现。这样可以在不完全重写旧机器人的情况下，持续增强当前能力。

当前覆盖层主要负责：

- 统一配置窗口。
- 飞书通知。
- 图片下载后处理。
- 产品信息图片清空和本地重传。
- 产品描述图片模块清空和本地重传。
- 仓库模板、仓库自检、仓库最终校验。
- 停止按钮和停止事件。
- 关闭窗口时清理子进程，避免 exe 残留。
- 直接发布和人工审核发布两种模式。
- 打包运行包时保留用户配置。

## 5. 配置文件原理

项目把配置保存在运行目录的 `work/state` 下。

机器人主要配置文件：

```text
dist/DxmTemuTerminalRobot/work/state/automation-config.json
```

AI API 配置文件：

```text
dist/DxmTemuTerminalRobot/work/state/laozhang-api.json
```

源码目录中的默认配置也会存在于：

```text
work/state/automation-config.json
work/state/laozhang-api.json
```

打包脚本会复制 `work` 到 `dist/DxmTemuTerminalRobot/work`，并特别保留运行包里已有的 `work/state/*.json`。这样重新打包时不会覆盖你已经保存过的 API Key、webhook、仓库、运费模板、图片目录等用户配置。

`automation-config.json` 中通常包含：

- 图片根目录。
- 批量处理数量。
- 是否每次运行自动新建子目录。
- 是否填写完成后直接发布。
- 仓库名称。
- 仓库模板列表。
- 运费模板。
- 承诺发货天数。
- 图片处理配置。
- 飞书机器人配置。

`laozhang-api.json` 中通常包含：

- 服务商名称。
- Base URL。
- API Key。
- 模型名称。
- 超时时间。

原则上不要把 webhook 和 API Key 写死进源码，应该只保存在本地配置文件中。

## 6. 控制面板原理

机器人启动后会显示 Tk 控制面板。控制面板是当前用户和机器人交互的中心。

主要能力包括：

- 设置仓库、运费模板、发货天数。
- 设置图片目录和批量数量。
- 设置是否直接发布。
- 设置是否启用图片下载后处理。
- 设置 AI API。
- 设置飞书 webhook。
- 测试飞书通知。
- 开始自动流程。
- 确认发布已编辑页。
- 打开手动菜单。
- 停止当前任务。
- 退出程序。

控制面板保存配置后，会写入 `automation-config.json` 和 `laozhang-api.json`。下一次打开程序时应该从这些文件恢复上次保存的配置。

## 7. 自动流程原理

一个商品的完整自动流程可以概括为：

1. 打开或绑定当前店小秘 Temu 编辑页。
2. 扫描页面模块，识别基本信息、产品信息、变种信息、产品描述、运输信息。
3. 补全基本信息必填属性。
4. 修正产品标题和英文标题。
5. 下载产品信息模块图片。
6. 删除页面上已经下载的原图。
7. 对下载图片进行本地处理。
8. 把处理后的本地图片重新上传到产品信息模块。
9. 填写变种信息。
10. 处理产品描述中的图片模块。
11. 填写运输信息。
12. 根据配置选择人工审核或直接发布。

流程执行时会持续写日志，日志会显示每一步动作、成功、警告、失败原因和重试情况。

## 8. 图片处理原理

图片处理分为产品信息图片和产品描述图片两块。

### 8.1 产品信息图片

产品信息图片的目标流程是：

1. 从店小秘 `产品信息` 模块读取图片地址。
2. 下载图片到本地目录。
3. 生成 `manifest.json`，记录来源链接、保存路径、失败原因等。
4. 删除页面上已经成功下载的原图。
5. 对本地图片进行后处理。
6. 点击页面的 `本地图片` 上传入口。
7. 通过 Playwright file chooser 或 input fallback 上传处理后的本地图片。

图片后处理的常见规则：

- 转为 1:1 方图。
- 目标尺寸默认 800 x 800。
- 背景默认白色。
- 小图会被移入 `_ignored_small_images`，避免误传。
- 生成 `image-postprocess-manifest.json` 记录处理结果。
- 最多上传店小秘允许的图片数量。

### 8.2 产品描述图片

产品描述图片的目标流程是：

1. 打开 `产品描述` 编辑器。
2. 检查当前是否存在图片模块。
3. 如果有图片模块，执行 `批量操作 > 清空描述 > 清空图片模块`。
4. 如果没有图片模块，跳过清空，直接上传。
5. 执行 `批量操作 > 批量传图 > 选择图片 > 本地图片`。
6. 上传本地图片。
7. 保存产品描述编辑器。

这个逻辑的关键点是：不是逐张删除，而是优先用店小秘自带的“清空图片模块”功能，再用本地图片重新上传。这样更接近人工操作，也更稳定。

## 9. 仓库选择原理

仓库是当前项目最容易出错的区域之一，所以机器人采用“配置文本框 + 模板 + 页面自检 + 精确校验”的策略。

### 9.1 仓库配置来源

仓库来源有三种：

1. 主窗口文本框中的仓库配置。
2. `automation-config.json` 中保存的仓库配置。
3. 仓库自检弹窗中用户重新选择并保存的模板。

多个仓库可以用逗号、顿号、分号或换行分隔。

### 9.2 仓库名称标准化

店小秘页面里经常出现：

```text
美中仓库
美中仓库（其他）
美中仓库(其他)
```

机器人会把 `(其他)`、`（其他）`、`(其它)`、`（其它）` 作为仓库标签后缀处理。也就是说，配置里的 `美中仓库` 可以对应页面里的 `美中仓库（其他）`。

这个匹配不是随意模糊匹配。它的核心原则是：

- 去掉“其他/其它”这类上下文后缀后再比较。
- 不允许把完全不同的仓库误当成同一个仓库。
- 页面最终校验时仍然要求目标仓库全部选中，模板外仓库不能被选中。

例如：

```text
配置：美中仓库，美西仓库，美东仓库
页面：美中仓库（其他），美西仓库（其他），美东仓库（其他）
```

这种可以匹配。

但如果页面额外选中了：

```text
煊烊-美东仓（其他）
```

它不属于配置目标，最终校验应该取消它，或者在无法取消时直接报错。

### 9.3 仓库自检弹窗

当配置仓库在页面上没有匹配成功，或者页面最终校验失败时，机器人会打开仓库自检弹窗。

弹窗会显示：

- 原配置。
- 失败原因。
- 页面实际读取到的仓库候选项。
- 可多选列表。
- 手动补充输入框。
- 是否保存为模板。
- 模板名。
- 保存配置并继续。

用户选择后，机器人会立即做三件事：

1. 去重并标准化仓库名。
2. 回填到主窗口的“选择仓库”文本框。
3. 写入 `automation-config.json`。

随后机器人继续使用原来的多仓库选择逻辑执行页面点选，而不是改成一套完全不同的新逻辑。

### 9.4 最终校验逻辑

仓库最终校验分两部分：

1. 目标仓库必须全部选中。
2. 模板外仓库必须全部取消。

机器人会打开仓库下拉框，清空搜索框，从完整列表读取当前选中状态。然后：

- 对模板外已经选中的仓库执行真实点击取消。
- 对模板内未选中的仓库执行原有选择逻辑。
- 再次读取页面选中状态。
- 如果仍然存在少选或多选，则报错并触发仓库自检。

最终日志中会显示：

```text
仓库最终校验通过
```

或者显示：

```text
仓库最终校验失败：缺少目标仓库=...；模板外仓库=...
```

这样可以防止“只配置了三个仓库，页面却残留第四个仓库”的问题。

## 10. 发布流程原理

发布逻辑分为两种模式。

### 10.1 人工审核模式

如果没有勾选“填写完成后直接发布”，机器人只负责把商品编辑好。

每个编辑好的页面会记录到：

```text
work/state/edited-pages.json
```

用户人工检查后，再点击“确认发布已编辑页”。发布时：

- 已发布的记录会跳过。
- 已关闭或找不到页面的记录会标记为 skipped。
- 单个商品发布失败会跳过，不影响后续商品。
- 最终写入发布进度和结果。

### 10.2 直接发布模式

如果勾选“填写完成后直接发布”，每个商品页面填写完成后立即尝试发布。

直接发布模式的原则是：

- 一个商品填完就发布，不等待全部商品完成。
- 发布失败要记录错误。
- 发布失败不应该让整个流程卡死在当前页面。
- 需要时通过飞书通知错误或停止状态。

这适合批量任务里允许跳过异常商品的场景。

## 11. 停止与退出原理

机器人内部使用 `STOP_EVENT` 表示停止请求。

点击“停止当前任务”时：

- 设置 `STOP_EVENT`。
- 当前自动化步骤在下一个检查点主动停止。
- 日志显示停止请求已发送。
- 不强制关闭当前店小秘页面。
- 不强制关闭 Chrome。

这样做是为了保留当前页面状态，方便用户人工查看或继续处理。

关闭机器人窗口或退出程序时，规则不同：

- 程序会清理自己启动的子进程。
- 程序会清理锁文件。
- 尽量避免 `DxmTemuTerminalRobot.exe` 残留。
- 不把手动打开的页面关闭动作当成普通停止动作的一部分。

也就是说：

- “停止任务”是停止自动化，不等于关闭页面。
- “关闭窗口/退出程序”是结束机器人进程，必须避免进程残留。

## 12. 飞书通知原理

飞书通知配置在统一配置窗口中。

核心字段：

- 是否启用飞书通知。
- Webhook。
- 签名 Secret。
- 关键词。
- 错误通知。
- 停止通知。
- 成功通知。

当 webhook 为空时，机器人会自动关闭飞书能力，不发送通知。

飞书通知主要用于：

- 流程错误。
- 用户停止任务。
- 任务完成。
- 测试通知。

安全原则：

- webhook 不写入源码。
- secret 不写入源码。
- webhook 为空时不启用飞书。

## 13. 打包与运行包原理

机器人打包脚本：

```text
scripts/package_terminal_exe.py
```

打包输出：

```text
dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe
```

打包流程：

1. 检查 `work/extracted_terminal_app.pyc` 是否存在。
2. 检查 `work/terminal_app_base.pyc` 是否存在。
3. 删除旧的临时 build 和 root exe。
4. 使用 PyInstaller `--onefile` 打包 `terminal_app.py`。
5. 收集 Playwright、tkinter、filedialog、messagebox 等运行依赖。
6. 把基座 pyc 作为 add-data 嵌入 exe。
7. 把生成的 exe 移动到 `dist/DxmTemuTerminalRobot`。
8. 复制源码目录的 `work` 到运行包。
9. 忽略 Chrome 缓存、日志、下载目录等无用大文件。
10. 保留运行包已有的 `work/state/*.json`，避免覆盖用户配置。

`dist` 中允许保留：

- 当前交付 exe。
- 当前交付运行包。
- 必要 zip 包。
- 其他机器人 exe，例如 `YunqiCollectorRobot.exe`。

不建议保留：

- 根目录临时 exe。
- PyInstaller `build` 目录。
- 临时 `.spec`。
- `__pycache__`。
- Chrome Cache、GPUCache、Code Cache、Crashpad 等缓存。

## 14. 前端、后端与插件的关系

虽然 `DxmTemuTerminalRobot` 是当前重点，但整个仓库还包括前端、后端和浏览器插件。

### 14.1 前端工作台

`frontend` 主要用于商品选择、规则配置、结果查看和操作入口。它适合承载可视化工作流。

### 14.2 后端模块

`backend` 主要用于数据采集、批量导出、筛选配置、Yunqi 相关任务和测试。

### 14.3 浏览器插件

`browser_extension` 用于浏览器内采集和页面辅助能力。它可以和前端/后端数据流配合，也可以为后续页面信号采集提供基础。

### 14.4 桌面机器人

`terminal_app.py` 和 `dist/DxmTemuTerminalRobot` 负责真实操作店小秘页面。它更像执行层，直接对网页进行点击、输入、上传和发布。

这几部分不是互相替代，而是分工：

- 前端负责管理和展示。
- 后端负责数据加工。
- 插件负责浏览器页面采集。
- 桌面机器人负责网页自动化执行。

## 15. 日志与状态文件

机器人会输出实时日志，也会把部分关键结果保存为 JSON。

常见状态文件包括：

- 图片下载结果。
- 图片后处理 manifest。
- 产品信息图片替换结果。
- 产品描述图片模块处理结果。
- 仓库自检结果。
- 发布进度。
- 已编辑页面记录。

这些状态文件的作用是：

- 方便排查失败原因。
- 方便任务中断后人工判断当前进度。
- 方便后续跳过已成功页面或已发布页面。

## 16. 维护原则

后续维护这个项目时，建议遵守以下原则。

### 16.1 配置和源码分离

API Key、webhook、仓库模板、图片目录等都属于用户配置，不应该写死到源码里。

应该保存到：

```text
work/state/*.json
dist/DxmTemuTerminalRobot/work/state/*.json
```

### 16.2 自动化逻辑尽量贴近人工操作

店小秘页面是动态前端页面，直接改 DOM 很容易造成页面状态不同步。机器人应该优先使用：

- 真实点击。
- 真实输入。
- 下拉框实际打开后读取候选。
- 文件选择器或 input 上传。
- 页面状态变化作为等待条件。

### 16.3 不要依赖固定等待

固定 `sleep` 会让速度越来越慢，也容易在网络慢时失败。更好的方式是等待：

- 模块出现。
- 图片数量变化。
- 编辑器打开。
- 保存按钮消失或状态变化。
- 下拉候选出现。
- 选中状态变化。

### 16.4 停止任务和关闭程序要分开

停止任务只应该停止自动化流程，不应该擅自关闭用户正在看的页面。

关闭程序才需要清理机器人自己的子进程，避免 exe 残留。

### 16.5 仓库逻辑必须最终精确校验

仓库可以做有限标准化，例如处理 `（其他）` 后缀，但不能只靠模糊匹配结束。

最终一定要校验：

- 配置内仓库全部选中。
- 配置外仓库全部取消。

### 16.6 图片流程必须可人工介入

图片处理的合理顺序是：

1. 先下载。
2. 再删除页面原图。
3. 用户或程序本地处理图片。
4. 再上传本地图片。

这样用户可以在图片编辑器中介入修改图片，不会被机器人直接覆盖。

## 17. 不要删除的关键文件

以下文件或目录通常不要删除：

```text
work/chrome-profile
work/state
work/terminal_app_base.pyc
work/extracted_terminal_app.pyc
dist/DxmTemuTerminalRobot/work/state
storage
.env
```

原因：

- `work/chrome-profile` 保存登录态。
- `work/state` 保存配置和运行状态。
- `terminal_app_base.pyc` 和 `extracted_terminal_app.pyc` 是机器人基座。
- `dist/DxmTemuTerminalRobot/work/state` 是已打包运行包的用户配置。
- `storage` 是业务数据。
- `.env` 可能包含本地环境配置。

## 18. 可以清理的常见文件

以下内容一般可以在确认没有运行中的打包任务后清理：

```text
build
__pycache__
*.spec 临时文件
根目录临时 DxmTemuTerminalRobot.exe
Chrome Cache
Code Cache
GPUCache
Crashpad
BrowserMetrics
downloads 临时下载目录
*.tmp
*.log
```

清理前要注意：

- 不要删除 `dist/DxmTemuTerminalRobot/DxmTemuTerminalRobot.exe`。
- 不要删除 `dist/DxmTemuTerminalRobot/work/state/*.json`。
- 不要删除正在使用的图片目录。
- 不要删除 `storage` 中的业务数据。

## 19. 一句话总结

这个项目的本质是：用本地配置和可控的 Python/Playwright 自动化，把店小秘 Temu 商品编辑页上的重复操作变成可停止、可恢复、可人工审核、可通知、可打包交付的桌面机器人流程。
