## 1. 项目定位

`Temu 选品上架工作台` 是一个内部运营使用的网页工作台，不是普通商城前台，也不是数据分析看板。

系统面向 Temu 运营人员，核心任务是把云启数据中的商品机会转化为可准备上架的商品资料。运营人员通过云启数据发现商品机会后，在系统中完成商品清洗、商品查看、1688 找货、成本录入、利润核算、上架判断和店小蜜批量导入模板生成。

核心定位可以概括为：

```text
云启商品机会发现
↓
1688 找货和成本确认
↓
利润核算和上架判断
↓
店小蜜批量导入模板生成
```

系统第一阶段强调半自动化，而不是全自动化。人的判断仍然保留在关键环节，例如选择商品、确认 1688 候选货源、录入采购价和判断商品是否适合上架。

### 1.1 不是普通商城前台

本系统不面向消费者，不提供商品浏览、购物车、支付、订单、会员等商城前台能力。页面设计应以运营效率为中心，而不是营销展示。

### 1.2 不是 BI 数据看板

本系统会展示部分数据指标，例如上传批次、待处理商品数、已核价商品数和可上架商品数，但这些指标服务于业务流程推进。系统重点不是搭建复杂数据分析看板，也不是做多维报表分析。

### 1.3 是内部运营工作流系统

系统的核心价值在于把分散在 CSV、Excel、1688、人工核价、店小蜜模板中的流程串起来，让运营人员能在一个网页工作台中持续推进选品上架任务。

## 2. 第一阶段业务范围

第一阶段只做最基础、最核心的业务链路：

```text
云启 CSV/Excel 上传
↓
数据读取、清洗、标准化
↓
商品列表回显
↓
选择一个商品
↓
去 1688 找替代品 / 同款 / 相似款
↓
录入或抓取 1688 成本信息
↓
利润计算
↓
判断是否可以上架
↓
生成店小蜜批量导入模板 Excel
```

### 2.1 第一阶段要做

第一阶段包含以下能力：

- 云启 CSV/Excel 上传
- 云启字段识别
- 商品数据清洗
- 商品数据回显
- 商品状态管理
- 1688 搜索词生成
- 1688 找货记录
- 手动录入 1688 采购价、运费、重量、MOQ
- 利润计算
- 判断是否建议上架
- 店小蜜批量导入 Excel 模板生成
- 文件下载
- 基础任务记录

### 2.2 第一阶段不做

第一阶段明确不做以下能力：

- 同类 SKU 组合推荐
- AI 自动选品
- 自动生成商品图
- 自动发布到 Temu
- 上架后自动核价
- 复杂权限系统
- 多租户 SaaS
- 大型推荐模型

### 2.3 第一阶段原则

第一阶段应遵守以下原则：

- 先打通完整主链路，再扩展自动化能力
- 先保留人工确认入口，再逐步引入 RPA 和 AI
- 先保证数据结构清晰，再追求复杂推荐能力
- 先服务个人或小团队本地使用，再考虑多人协作和部署
- 页面设计以操作效率为核心，不做营销式页面

## 4. 为什么不用 Streamlit

Streamlit 适合快速 MVP，尤其适合在早期快速验证数据读取、数据清洗、表格展示、简单表单和 Excel 导出流程。旧项目已经证明，Streamlit 很适合用于业务流程原型验证。

但是本项目的新目标是做一个长期维护的正式网页系统，Streamlit 不适合作为长期主框架。

### 4.1 页面复杂后维护困难

第一阶段虽然范围克制，但页面已经包含上传、字段匹配、商品表格、详情抽屉、找货记录、利润核算、状态管理、导出记录和任务中心。后续还会加入 RPA、AI、多人协作和更多状态流转。

在 Streamlit 中，复杂页面通常会逐渐变成大量脚本式代码。随着状态和交互变多，页面结构、业务逻辑和数据处理很容易混在一起。

### 4.2 前后端逻辑容易混在一起

Streamlit 默认倾向于把 UI、数据读取、业务计算和状态处理写在同一套 Python 脚本里。对于 MVP 这很方便，但对于长期项目会造成模块边界不清晰。

本项目需要清晰地区分：

- 前端展示和交互
- 后端 API
- 数据清洗模块
- 利润计算模块
- 店小蜜模板导出模块
- RPA Worker
- 配置文件

这种结构更适合前后端分离架构。

### 4.3 复杂操作交互不够灵活

本项目需要大量后台操作类交互，例如：

- 商品表格筛选
- 表格行状态标签
- 商品详情抽屉
- 找货候选卡片
- 利润核算表单
- 批量选择
- 模板字段完整度检查
- 任务进度和错误提示

这些交互用 React + Ant Design 会更自然，也更容易长期维护。

### 4.4 不适合长期维护 RPA 任务中心

后续 RPA 任务可能包含：

- 1688 搜索
- 1688 图搜
- 1688 详情页抓取
- 店小蜜页面辅助操作
- 截图保存
- 错误日志
- 失败重试

这些任务需要排队、状态记录、错误追踪和结果回写，不应该被塞进 Streamlit 页面执行流程里。

### 4.5 不适合模块化管理核心业务域

本项目至少包含以下业务域：

- 云启导入
- 商品标准化
- 1688 找货
- 利润核算
- 上架判断
- 店小蜜模板导出
- RPA 任务
- 配置管理

这些模块需要独立组织、独立测试和持续演进。FastAPI + React 的结构更适合做清晰的模块划分。

### 4.6 不适合后续团队内部多人使用

如果后续团队内部多人使用，需要更好的页面组织、API 边界、数据库设计、日志记录和部署方式。Streamlit 仍可以作为内部小工具存在，但不适合作为这个正式工作台的主框架。

### 4.7 旧项目不废弃

旧 Streamlit 项目不应直接废弃。它仍然是重要的业务参考，尤其是：

- 云启文件读取经验
- CSV 编码处理经验
- 字段清洗规则
- 利润计算初版逻辑
- Excel 导出经验
- 运营流程理解

新项目应参考旧项目中的业务逻辑和规则经验，但不要照搬旧项目的脚本式组织方式。

## 5. 为什么采用 React + TypeScript + Vite + Ant Design

React + TypeScript + Vite + Ant Design 适合搭建正式的前端运营工作台。

### 5.1 React 适合复杂交互工作台

本项目的页面不是简单表单，而是一个多流程、多状态、多组件的操作系统。React 适合把页面拆成可维护组件，例如：

- 商品表格
- 商品详情抽屉
- 1688 候选货源卡片
- 利润核算面板
- 工作流步骤条
- 任务状态列表
- 字段映射设置表单

这些组件可以逐步复用和扩展。

### 5.2 TypeScript 降低长期维护风险

本项目涉及大量结构化数据：

- 云启原始字段
- 标准化商品字段
- 1688 候选货源
- 利润记录
- 上架候选记录
- 导出记录
- 任务状态

TypeScript 可以让前端数据结构更清晰，减少字段拼写错误、接口变更遗漏和状态判断错误。

### 5.3 Vite 适合快速开发

Vite 启动快、热更新快，适合本地开发阶段快速迭代。第一阶段本地开发运行即可，不需要一开始引入复杂前端工程配置。

### 5.4 Ant Design 适合后台类系统

Ant Design 对后台运营系统非常友好。本项目会大量使用：

- Layout
- Menu
- Table
- Upload
- Form
- Drawer
- Modal
- Tabs
- Steps
- Tag
- Progress
- Alert
- Card

这些组件能够快速搭建一个清晰、稳定、可维护的运营工作台。

### 5.5 前端主要职责

前端主要负责：

- 文件上传界面
- 商品列表回显
- 商品详情抽屉
- 1688 找货候选展示
- 利润核算表单
- 上架状态标签
- 导出模板操作
- 任务进度展示

前端不应承担 CSV/Excel 深度解析、利润规则核心计算、Excel 模板生成和 RPA 执行等后端职责。

### 5.6 Axios 或 fetch

第一阶段可以使用 Axios 或原生 fetch。建议如果接口数量较多，使用 Axios 并在 `frontend/src/api/` 下按业务域封装：

- `uploadApi.ts`
- `productApi.ts`
- `sourcingApi.ts`
- `profitApi.ts`
- `exportApi.ts`

这样前端页面不直接散落 API URL，后续接口调整更方便。

## 6. 为什么采用 FastAPI

后端核心处理都在 Python 侧。FastAPI 适合做前后端分离 API，同时保留 Python 生态在数据处理、Excel、RPA 和 AI 接入方面的优势。

### 6.1 后端核心处理适合放在 Python

本项目后端需要处理：

- CSV/Excel 读取
- 云启字段映射
- 数据清洗
- 1688 搜索词生成
- 利润计算
- 店小蜜 Excel 模板生成
- 后续 RPA
- 后续 AI 接入

这些能力都非常适合 Python 生态。

### 6.2 FastAPI 适合前后端分离

FastAPI 可以提供清晰的 REST API，供 React 前端调用。后端可以按业务域拆分路由：

- 上传路由
- 商品路由
- 1688 找货路由
- 利润核算路由
- 导出路由
- 任务路由

这种结构比把所有逻辑写在单个脚本中更适合长期维护。

### 6.3 Pydantic 定义清晰数据结构

Pydantic 用于定义请求和响应结构，例如：

- 商品详情响应
- 商品列表筛选参数
- 1688 候选货源提交参数
- 利润核算请求
- 利润核算结果
- 导出任务响应
- RPA 任务状态

这有助于前后端对齐字段，并减少隐藏的结构不一致。

### 6.4 后端主要职责

后端主要负责：

- 接收上传文件
- 解析云启数据
- 保存商品数据
- 管理商品状态
- 保存 1688 找货记录
- 执行利润计算
- 生成店小蜜 Excel
- 管理任务状态

### 6.5 Pandas、openpyxl、PyYAML 的作用

Pandas：

- 读取 CSV/Excel
- 清洗表格数据
- 处理字段标准化
- 做基础数据转换

openpyxl：

- 读取或写入 Excel 模板
- 生成店小蜜批量导入 Excel
- 保留模板列结构

PyYAML：

- 读取字段映射配置
- 读取利润规则配置
- 读取店小蜜模板字段映射配置
- 读取 RPA 设置

## 7. 为什么数据库先用 SQLite

第一阶段系统主要是个人或小团队本地使用，SQLite 足够。

### 7.1 SQLite 适合第一阶段

SQLite 的优点：

- 无需单独安装数据库服务
- 本地开发启动简单
- 文件型数据库便于备份
- 对第一阶段数据量足够
- 与 SQLAlchemy 配合后，后续迁移到 PostgreSQL 成本较低

第一阶段的重点是把业务链路跑通，而不是先搭建复杂数据库运维体系。

### 7.2 SQLite 保存的数据

SQLite 用来保存：

- 上传批次
- 商品数据
- 1688 候选货源
- 利润核算记录
- 上架候选记录
- 导出记录
- RPA 任务记录

### 7.3 后续迁移 PostgreSQL

当出现以下情况时，可以考虑迁移到 PostgreSQL：

- 多人同时使用
- 部署到局域网或服务器
- 数据量明显增长
- 需要更强并发能力
- 需要更复杂查询和索引
- 需要更稳定的备份和恢复机制

迁移前应确保 SQLAlchemy 模型、Alembic 迁移和数据库访问层边界清晰。

## 8. 为什么 RPA 独立成 Worker

RPA 不应该直接写在 API 路由里。API 路由应保持快速响应，只负责创建任务、查询状态和读取结果。

### 8.1 RPA 任务的特点

RPA 任务通常具有以下特点：

- 1688 页面操作可能耗时
- 浏览器自动化可能失败
- 页面结构可能变化
- 登录状态可能失效
- 网络和加载时间不稳定
- 需要记录截图、错误日志、重试状态
- 不能阻塞后端接口
- 后续可能多个 RPA 任务排队执行

因此 RPA 不适合直接在 FastAPI 路由函数里同步执行。

### 8.2 Worker 目录规划

RPA 应该放在：

```text
backend/workers/
```

例如：

```text
backend/workers/rpa_1688_worker.py
backend/workers/rpa_dianxiaomi_worker.py
```

### 8.3 API 和 Worker 的分工

后端 API 负责：

- 创建 RPA 任务
- 写入 `rpa_tasks`
- 返回任务 ID
- 查询任务状态
- 读取任务结果
- 触发失败任务重试

Worker 负责：

- 拉取待执行任务
- 启动 Playwright
- 执行页面操作
- 保存截图
- 保存错误原因
- 更新任务状态
- 回写抓取结果

### 8.4 第一阶段 RPA 策略

第一阶段先保留 RPA 接口和任务结构，不强制实现全自动 RPA。找货可以先通过：

- 生成 1688 搜索词
- 打开 1688 搜索链接
- 运营人员手动查看
- 手动录入候选货源

后续再逐步增强 Playwright 自动搜索、图搜和详情页抓取。

### 9.1 前端目录说明

`frontend/src/pages/` 放页面级组件，每个文件对应一个主要页面。

`frontend/src/components/` 放可复用业务组件，例如商品表格、商品详情抽屉、候选货源卡片和利润核算面板。

`frontend/src/api/` 放前端 API 封装，页面不直接硬编码后端接口地址。

### 9.2 后端目录说明

`backend/app/api/` 放 FastAPI 路由。路由只负责请求参数处理、调用服务模块和返回响应，不堆积复杂业务逻辑。

`backend/app/models/` 放 SQLAlchemy 模型。

`backend/app/schemas/` 放 Pydantic 请求和响应结构。

`backend/app/modules/` 放业务模块。云启、1688、利润、导出分别拆分，避免一个大文件承担所有逻辑。

`backend/workers/` 放异步任务和 RPA Worker。

### 9.3 配置和存储目录说明

`configs/` 放 YAML 配置文件，避免把字段映射、模板列名和利润规则写死在代码里。

`storage/` 放运行期文件：

- 原始上传文件
- 清洗后中间文件
- 导出的 Excel
- 店小蜜模板
- RPA 截图

## 11. 核心数据流

第一阶段核心数据流如下：

```text
上传云启 CSV
↓
保存 upload_batch
↓
解析为标准 product 数据
↓
商品进入未处理状态
↓
用户选择商品进行 1688 找货
↓
保存 sourcing_candidate
↓
用户录入采购价、运费、重量等成本
↓
生成 profit_record
↓
系统判断是否建议上架
↓
通过的商品进入 listing_candidate
↓
生成店小蜜 Excel
↓
保存 export_record
```

### 11.1 上传和解析

用户在 `YunqiImport` 页面上传云启 CSV/Excel。后端保存原始文件到 `storage/uploads/`，创建 `upload_batch` 记录。

后端读取文件，识别编码和字段，通过 `configs/yunqi_field_mapping.yaml` 把云启原始字段映射为系统标准字段。

清洗后的商品数据保存到 `products` 表，初始状态为 `imported`。

### 11.2 商品查看和找货

用户在 `ProductReview` 页面查看商品列表。查看过的商品可标记为 `reviewed`。当用户选择商品进入 1688 找货流程时，商品状态可变为 `sourcing`。

系统根据商品标题、分类和关键词生成 1688 搜索词。用户可以修改搜索词，并打开 1688 搜索链接。

用户手动录入候选货源后，系统保存到 `sourcing_candidates` 表。若至少有有效候选货源，商品状态可变为 `sourced`。

### 11.3 利润核算和上架判断

用户选择一个候选货源进入利润核算。系统根据 Temu 预估售价、采购价、运费、重量、包装成本、损耗比例、平台费用和汇率计算利润。

利润核算结果保存到 `profit_records` 表。完成核价后，商品状态可变为 `priced`。

如果利润率和风险规则满足配置要求，系统创建 `listing_candidate` 记录，商品状态变为 `approved`。如果不满足要求，商品状态变为 `rejected`。

### 11.4 模板导出

用户在 `ListingExport` 页面选择已通过利润核算的商品。系统检查店小蜜模板字段完整度，根据 `configs/dianxiaomi_template_mapping.yaml` 生成 Excel。

生成的文件保存到 `storage/exports/`，并创建 `export_record` 记录。导出成功后，相关商品状态可变为 `exported`。

## 12. 商品状态设计

商品状态用于记录商品在选品上架流程中的当前位置。

建议状态如下：

| 状态值 | 中文含义 | 说明 |
| --- | --- | --- |
| imported | 已导入 | 商品从云启 CSV/Excel 导入并完成标准化，尚未人工查看 |
| reviewed | 已查看 | 运营人员已经查看过商品详情 |
| sourcing | 找货中 | 商品已进入 1688 找货流程 |
| sourced | 已找到货源 | 已保存至少一个 1688 候选货源 |
| priced | 已核价 | 已完成利润核算 |
| approved | 建议上架 | 利润和规则判断通过，可进入导出清单 |
| rejected | 不建议上架 | 利润或风险规则不通过 |
| exported | 已导出 | 已生成店小蜜导入模板 |

### 12.1 状态流转建议

推荐状态流转如下：

```text
imported
↓
reviewed
↓
sourcing
↓
sourced
↓
priced
↓
approved / rejected
↓
exported
```

### 12.2 状态设计原则

- 状态应由后端统一维护，前端只展示和触发操作
- 状态变更应记录关键时间或操作记录
- 不建议在前端硬编码状态中文文案，前后端可以共享状态枚举或由后端返回展示文案
- 允许运营人员在特殊情况下重新找货或重新核价
- `rejected` 不代表永久删除，只代表当前条件下不建议上架

## 14. 配置文件设计

第一阶段需要使用配置文件，而不是把字段写死在代码里。

### 14.1 为什么要使用配置文件

原因：

- 云启导出的字段名可能变化
- 不同 CSV/Excel 文件可能存在字段别名
- 店小蜜模板列可能调整
- 利润规则会随着运营策略变化
- RPA 参数可能根据电脑环境变化
- 配置变化不应该频繁修改核心代码

配置文件让系统的业务规则更可控，也更方便后续由运营或开发者调整。

- 所有核心模块边界清晰，方便后续继续开发

### 15.1 模块边界验收

除功能可用外，还应检查模块边界：

- 前端页面不直接处理复杂 CSV/Excel 解析
- 后端路由不堆积大量业务细节
- 云启导入、1688 找货、利润核算和店小蜜导出分别有独立模块
- RPA 相关逻辑不直接写在 API 路由中
- 字段映射和利润规则不硬编码在核心逻辑里

### 15.2 数据验收

需要抽样确认：

- 原始上传文件能追溯到上传批次
- 每个商品能追溯来源行
- 商品状态流转符合流程
- 1688 候选货源能关联到商品
- 利润记录能关联到商品和候选货源
- 导出记录能追溯导出的商品清单

### 15.3 用户操作验收

运营人员应能完成以下闭环：

```text
上传云启文件
↓
查看商品列表
↓
选择商品找货
↓
打开 1688 搜索
↓
录入候选货源
↓
核算利润
↓
加入上架候选
↓
生成店小蜜 Excel
↓
下载 Excel
```
